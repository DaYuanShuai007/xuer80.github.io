Git是Android开发必不可少的版本控制工具，本人用了git之后，完全抛弃了svn，哈哈，Git教程网上很多，我基本是自己摸索的，我觉得廖雪峰 的网站Git教程讲的最浅显易懂，现在把我当初学习的笔记分享下咯~

Git是目前世界上最先进的分布式版本控制系统。版本控制是一种记录一个或若干文件内容变化，以便将来查阅特定版本修订情况的系统。

  1. Git简史

    Linus1991创建开源linux，随着代码库增大，代码合并变得重要，于是2005年linus大神用了两周时间用C写了Git，这是有多牛掰。Git之后迅速成为最流行的分布式版本控制系统。

  2. 集中式VS分布式

    CVS和SVN都是集中式的版本控制系统。

    1. 集中式版本控制系统

      版本库是集中存放在中央服务器的，业务用自己的电脑，需先从中央服务器取得最新的版本，然后开始进行修改扩展等工作，完成后再推送给中央服务器。这项工作必须联网才行，这也限制了工作效率的提升。

    2. 分布式版本控制系统

      而分布式版本控制系统没有“中央服务器”，每个人的电脑上都是一个完整的版本库，因此没有了联网的需求，而且在团队协作时，可以互相看到对方修改的地方。

    和集中式版本控制系统比，分布式版本控制系统的安全性要高的多，因为团队每个人都有一个完整的版本库，而集中式版本控制系统的中央服务器一旦出了问题，就无法进行工作了。

    实际上，团队工作时，Git也需要有一台充当“中央服务器”的电脑，但作用仅仅是用来方便“交换”大家的修改，有利于协作开发。

    Git的优势不仅在于不必联网，其还具备强大的分支管理能力，在众多分布时管理软件中，Git依然是最快，最简单，也是最流行的。

  5. 把文件添加到版本库

    所有的版本控制系统，其实只能跟踪“文本文件”的改动，如txt文件，网页，所有的程序代码等，Git也不例外。而二进制文件，如图片、视频等，只能跟踪其大小的变动，内容改动则不知道(word也是二进制格式，同理)。因此如果真正使用版本控制系统，就要以纯文本方式编写文件。

    因为文本是有编码的，常用中文GBK，日文Shift_JIS编码等，建议统一UTF-8编码，可以被所有平台支持。windows自带记事本保存UTF-8文件，开头会添加0xefbbbf字符，会导致一些错误，避免使用。

  6. 查看仓库文件状态

    git status命令可以让我们时刻掌握仓库当前的状态，哪些文件被修改，以及是否提交都会显示出来。显示红色表示文件已经更改，但是没有add进仓库，显示绿色标识，表示文件已经添加进仓库，但没提交

    git diff 故名思意就是查看difference,即查看修改内容。显示格式是Unix通用的diff格式，增加的内容前会有+号。文件已经提交或添加进仓库则不会显示数据。

  7. 版本回退

    Git中，用HEAD表示当前版本，上个版本就是HEAD^，上上个版本就是HEAD^^，一次类推，也可以写作HEAD~100，HEAD指向的版本就是当前版本。

    git reset命令回退版本 git reset –hard commit_id

    此时再git log看下版本提交信息

    最新的版本已经看不到了，已经回退到此前的版本。如果再想返回之前的最新提交的版本，需要记得他的commit id，这样就可以定点返回

    如此就又返回到原来的版本啦。版本号也就是commit id没必要写全，前几位Git就可以自动查找。Git版本回退速度非常快，因为Git在内部有个指向当前版本的HEAD指针，当回退版本时，Git仅仅是把HEAD从指向“还是test”改为指向“test2”而已，然后顺便就把工作区的文件更新了，因此HEAD指向哪个版本号，就把当前版本定位在哪。

  8. 工作区和暂存区

    Git的版本库存了很多东西，其中最重要的就是stage(或叫index)的暂存区，还有Git自动创建的master，以及指向master的一个指针HEAD。

  10. 撤销修改

    如果修改add到了暂存区，还没有commit，可以用命令git reset HEAD [file]把暂存区的修改撤销掉，重新放回工作区(注意这个时候只是把暂存区的文件回退到工作区，工作区的文件内容修改并没有撤销，可以用git checkout 撤销修改，或者直接删除修改)。git reset命令既可以回退版本，也可以把暂存区的修改回退到工作区。用HEAD时，表示最新的版本。

    3. 分支管理策略

      一般合并分支时，如果可能，Git会用Fast forward模式，但这种模式删除分支后，会丢掉分支信息。如果强制禁用Ff模式，Git就会再merge时生成一个新的commit，这样，从分支历史上就可看出分支信息。

      实际开发中，应该按如下几个原则为进行分支管理：

      • master分支应该保持稳定，也就是仅用来发布新版本，平时尽量不再其上干活
      • 干活尽量再其他分支，比如sb分支上，该分支不稳定，可以持续开发，到版本发布时，再把sb分支何并到master上，再master分支发布稳定版本
      • 团队合作是，每个人都在sb分支上干活，每个人都有属于自己的分支，不定时往sb分支上合并就行了

      首先，可以试图用git push origin branch-name推送自己的修改

      其次，如果推送失败，则因为远程分支比你的本地分支更新，需要先用git pull试图合并；

      如果合并有冲突，则解决冲突，并再本地提交

      没有冲突或解决掉冲突后，再用git push origin branch-name推送就能成功

      若git pull 提示 “no tracking information”则说明本地分支和远程分支的链接关系没有创建，用命令git branch –set-upstream branch-name origin/brach-name.

      这就是团队工作模式

      注意：本地新建的分支如不推送到远程，对其他人就是不可见的

      本地创建和远程分支对应的分支使用 git checkout -b branch-name origin/branch-name，本地和远程分支名称最好一致

      建立本地分支和远程分支的关联，使用git branch –set-upstream branch-name origin/branch-name

      从远程抓取分支，使用git pull，如果有冲突，要先处理冲突

  15. 自定义Git

    Git实际上有很多配置，不止user.name和user.email

    比如让Git显示颜色，让命令更醒目

    git config –global color.ui true

    这样Git会适当显示不同颜色

    1. 忽略特殊文件

      有些文件必须有，但没必要提交，如一般的配置文件等

      我们可以再Git工作区创建一个特殊的.gitignore文件，然后把要忽略的文件名填进去，Git就会自动忽略这些文件

      不需要从头写该文件，github准备了各种配置文件，只需组合一下，就可以使用，配置文件可浏览链接https://github.com/github/gitignore

      忽略的原则是

      • 系统自动生成的文件，如缩略图等
      • 编译生成的中间文件，可执行文件等，比如java编译产生的.class文件
      • 带有敏感信息的配置文件，如存放口令的配置文件

      忽略文件配置好后，就是把.gitignore提交到Git就可以了，当然检验.gitignore的标准时git status命令是否有working directory clean.

      如果有时你需要添加某文件到Git，但发现无法添加，那么有可能该文件被.gitignore忽略了

      此时可以用-f强制添加

      git add -f sb.class

      如果角色忽略文件写的有问题，需要找出哪个规则写错了，可以用git check-ignore命令检查

      git check-ignore -v sb.class

      GIt会反馈.gitignore忽略了该文件的规则位置，然后我们可以根据需要修改规则

    2. 配置别名

      如果经常敲错命令怎么办，可以采用别名的方式，简化命令，比如git status改成git st

      git config –global alias.st status 这样就把git status改成了git st

      当然也可以改其他命令

      git config –global alias.co checkout

      git config –global alias.ci commit

      git config –global alias.br branch

      –global参数时全局参数，也就是说这些命令再这台电脑上通用

      也有人丧心病狂，把lg配置成了

      别名就在alias后面，要删除别名，直接把对应的行删除掉就好

      而当前用户的Git配置文件放在用户主目录下的隐藏文件.gitconfig中

      配置别名也可以直接修改这个文件，如果改错了，可以删掉文件重新通过命令配置
